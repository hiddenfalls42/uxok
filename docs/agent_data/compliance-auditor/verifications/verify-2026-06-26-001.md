## Fixed This Round

### L-002 — Hook-system opener reworded — PASS
`docs/explanation/hook-system.md:1-5` now opens: "**Hooks** are serialized extension
points — named call sites ... the system runs every registered handler in priority order
within the caller's own asyncio task, then returns their collected results. The caller
waits ...". This is accurate against `src/uxok/hooks/_system.py`: `execute` is `async`
(`_system.py:146`), delegating to `_execute_now`, which awaits `_execute_hook_safe`
serially in a `for` loop over the snapshot (`_system.py:210-211`) — i.e. handlers (which
may themselves be async, awaited at `_system.py:65`) run serially in the caller's task.
"serialized" no longer collides with Python's sync/async sense.

Residual watch-item (not a regression): line 13 retains "This is **synchronous**
request-response communication across plugin boundaries." This is the messaging sense
(caller blocks for return values), not the Python coroutine sense, and the original L-002
was scoped to the opener. Acceptable — noted only so a future pass knows it was seen, not
missed.

### L-005 — Tutorial glob wording aligned — PASS
`docs/tutorials/getting-started.md:169-170`: "`@event("counter.*")` matches any event
whose name begins with `counter.`, including multi-segment names like `counter.ops.reset`."
This now expresses the dot-spanning behaviour and is consistent with the canonical

**(c) Relocated cycle-detection rationale is accurate and not duplicated.**
The "why the guard is only reachable on reload" theory moved to
`docs/explanation/plugin-architecture.md:17`:
> "At initial registration this is passive — declaring a dependency on an unregistered peer
> fails immediately, so cycles cannot form. During hot reload ... the registry actively
> checks for cycles before committing the new edges and rolls back automatically on
> rejection."

Verified against `src/uxok/registry/impl.py`:
- `_check_circular_dependencies` (`impl.py:440-473`) is an iterative DFS that fires only if
  `plugin_id` is reachable from its proposed dependencies through existing edges.
- At initial registration (`add`, `impl.py:137-148`) the plugin is not yet in `_plugins`
  (added at line 148, *after* the check at 138) and its deps must already exist
  (`impl.py:131-135`). A brand-new node has no incoming edges, so the DFS can never reach
  it — the check is structurally incapable of firing. "passive ... cycles cannot form" is
  an accurate description of the *effect*.
- On the reload path (`swap_instance` → `_replace_dependency_edges`, `impl.py:263-293`) the
  old edges are detached first (279-281), the cycle check runs against the
  graph-as-it-will-be (284), and on `PluginError` the old edges are restored and the error
  re-raised (285-289). "actively checks ... before committing ... rolls back automatically"
  is exact.

Not duplicated: the how-to (§4, lines 75-90) now states only the mechanism and how to catch
it; the explanation owns the "why." Different altitudes, no restated rationale.

### V-001 — `how-to-manage-plugin-dependencies.md:77` says the cycle error carries the plugin *name*; it carries the *ID* — Low
**File:** `docs/how-to/how-to-manage-plugin-dependencies.md:77`
**Signal:** `Version mismatch (minor)`

The trimmed §4 states: "It raises `PluginError` with the **name** of the plugin whose
proposed edges would close the cycle." The actual message
(`src/uxok/registry/impl.py:466-468`) interpolates `plugin_id` — a `PluginId` (UUID) —
not the human name: `"Circular dependency detected: plugin {plugin_id} is reachable from
its own proposed dependencies"`. The example comment on line 88
(`# PluginError: Circular dependency detected`) matches the real prefix and is fine; only
the "with the name" clause on line 77 overstates what the exception surfaces. Low — does not
mislead a reader into broken code, and the catch pattern shown is correct. Recommend
changing "the name of the plugin" to "the plugin id" (or dropping the clause) on the next
touch; not a blocker for this round.
