## Executive Summary

A complete, confirmed-live secure-capability feature set has landed in the kernel with essentially zero public-facing teaching documentation. The default (`capability_access="open"`) behaviour is documented and correct, but the `declared` and `sealed` postures — and everything they pull in (`resolves`, runtime `requires` gating, `CapabilityAccessError`, `self.core` attenuation, the sealed `CapabilityFacet` and return guard, `Core.check_plugin`, the two new capability events, and the `kernel.lifecycle` grant) — appear nowhere in `tutorials/`, `how-to/`, or `explanation/`. None of this is inaccuracy in the existing pages; it is absence. Two existing pages are also now materially incomplete because `requires` quietly gained a second, runtime role they predate. The recommended remedy is three new pages and five updates, sequenced so the foundational explanation lands before the how-tos that lean on it. A single Info item records pre-existing RFC-citation leaks (one in a public page, several in docstrings) that point into the build-excluded `proposals/` tree.

**Total findings: 12** — 7 Critical, 0 High, 2 Medium, 0 Low, 3 Info

#### C-001 — `CoreConfig.capability_access` (open/declared/sealed) has no public documentation
**File:** `src/uxok/_config_validation.py:59-61`
**Signal:** `Undocumented public interface`

`capability_access` is a public `CoreConfig` field — the headline of the feature set, setting the security posture for all capability resolution — with three modes (`"open"` default, `"declared"`, `"sealed"`). It is validated against `{"open","declared","sealed"}` but documented nowhere in the public layer. Per the auditor severity tiers, an undocumented configuration option is Critical, and this one sits squarely in the directive's primary scope. It needs two homes: the canonical "why" (a new explanation page) and the "how" (a new how-to).

**Expected location per skill:**
- NEW `docs/explanation/secure-capability-access.md` — **Diátaxis: Explanation**, discursive density (as long as needed). The canonical system-level rationale: what each mode changes, why a posture axis exists separate from the three resolution policies, and the trust model (open trusts the whole graph; declared/sealed bound a plugin to its manifest). This page is the public layer's single home for this "why" — the per-mode rationale must live here and not be restated in `capability-system.md` (the public non-restatement rule; `capability-system.md` names the concept and links here). Covers, at a system level, the surfaces in C-002 through C-007 so the how-tos can link rather than re-explain.
- NEW `docs/how-to/how-to-lock-down-capability-access.md` — **Diátaxis: How-to**, tight density (200–600 words). Numbered steps to set `capability_access` and grant resolution. **Runnable example required:** a `Core(capability_access="sealed")`, one plugin that declares `requires`/`resolves` and resolves within its grant (succeeds), and one resolution outside the grant (raises `CapabilityAccessError`). Minimal example fits in ~25 lines.

**Code ref:** `src/uxok/_config_validation.py:59-61`; `src/uxok/core/_core_facet.py`

#### C-002 — `Plugin(resolves=…)` runtime resolution grant is untaught
**File:** `src/uxok/plugin/_base.py:75`
**Signal:** `Undocumented public interface`

`resolves` is a public `Plugin` constructor parameter — a set of capability names a plugin is authorized to resolve at runtime, distinct from `requires` (which is load-order). In `declared`/`sealed` mode a plugin may resolve only capabilities in `requires | resolves` (plus the reserved `kernel.dispatch` grant). It is normalized but deliberately not validated at registration (`_base.py:130-154`); enforcement happens at resolution via `enforce_requires`. No public page teaches the `requires` vs `resolves` distinction.

**Expected location per skill:**
- NEW `how-to-lock-down-capability-access.md` (C-001) is the primary teaching home — the grant is the mechanism the lock-down task configures.
- UPDATE `docs/how-to/how-to-work-with-capabilities.md` — **Diátaxis: How-to**, 200–600 words. Add a step distinguishing `requires` (load-order precondition) from `resolves` (runtime grant), pointing to the lock-down how-to for the secure-mode detail.
- Reference for the `resolves` parameter is **generated from the `_base.py` docstring** — the writer does not author it (see I-002); the docstring is already present at `_base.py:96`.

**Code ref:** `src/uxok/plugin/_base.py:75-154`

#### C-003 — Secure-mode binding model (`self.core` attenuation, sealed `CapabilityFacet`, sealed return guard, `StalePluginError`) is undocumented
**File:** `src/uxok/plugin/_base.py:269`, `src/uxok/core/_core_facet.py`
**Signal:** `Undocumented public interface`

A cluster of behaviours appears only under `declared`/`sealed` and is wholly undocumented:
- Under declared/sealed, `self.core` is a `CoreFacet` — a plugin-safe, manifest-bounded view of the kernel, not the real `Core` (`_base.py:269`).
- In `sealed` mode, a **typed** capability resolution returns a `CapabilityFacet` exposing only the protocol surface; it transparently rebinds across a provider swap and raises `StalePluginError` after revoke. **Untyped** resolutions return the raw provider.
- The sealed **return guard**: a sealed capability method that returns a live `Plugin`/`Core`/facet is refused with `CapabilityAccessError` — one-hop accidental-leak protection, explicitly **not** a security boundary.

`StalePluginError` is a public top-level export (`uxok.__all__`) with no teaching page. The "not a security boundary" qualifier is a tradeoff statement and belongs in explanation, not a how-to.

**Expected location per skill:**
- NEW `explanation/secure-capability-access.md` (C-001) — the canonical home. Cover the attenuation model, what the sealed facet hides, why the return guard is a guardrail and not a boundary (the "when this does not protect you" tradeoff the Explanation section is for).
- UPDATE `how-to-lock-down-capability-access.md` (C-001) — show catching `StalePluginError` after a provider is revoked in sealed mode.
- Reference for `CapabilityFacet`, `CoreFacet`, `StalePluginError` is **generated from docstrings** (see I-002).

**Code ref:** `src/uxok/plugin/_base.py:269`; `src/uxok/core/_core_facet.py`

#### C-004 — `CapabilityAccessError` is exported but untaught
**File:** `src/uxok/__init__.py`
**Signal:** `Undocumented public interface`

`CapabilityAccessError` is a top-level export from `uxok`. It is raised in two situations: when a plugin resolves a capability outside its `requires | resolves` grant, and when a sealed capability method returns a live authority handle (the return guard, C-003). No public page tells a reader when to expect it or how to handle it. A reader who turns on `declared`/`sealed` mode and follows the existing capability how-tos will hit this exception with no documented explanation.

**Expected location per skill:**
- NEW `how-to-lock-down-capability-access.md` (C-001) — the runnable example must trigger and catch it (resolution outside grant).
- NEW `explanation/secure-capability-access.md` (C-001) — explain the two raise conditions as part of the binding model.
- Reference is **generated from the exception's docstring** (see I-002).

**Code ref:** `src/uxok/__init__.py`; `src/uxok/core/_core_facet.py`

#### C-005 — `Core.check_plugin(candidate) -> AdmissionResult` admission probe is undocumented
**File:** `src/uxok/core/_core_facet.py` (and `_core.py:291` per directive)
**Signal:** `Undocumented public interface`

`Core.check_plugin` is a public, advisory, side-effect-free admission probe: it answers whether a candidate's manifest fits the live graph, returning an `AdmissionResult` with an `ok` flag plus per-fault detail. Neither `check_plugin` nor `AdmissionResult` is documented. This is a distinct task ("can I admit this plugin before I commit to registering it?") and warrants its own how-to rather than folding into the registration guide.

**Expected location per skill:**
- NEW `docs/how-to/how-to-probe-a-plugin-before-admission.md` — **Diátaxis: How-to**, 200–600 words. Numbered steps: build a candidate, call `check_plugin`, branch on `AdmissionResult.ok`, read per-fault detail. **Runnable example required:** a core with one provider registered, then `check_plugin` on a candidate that requires a missing capability (returns `ok=False` with the fault) versus one that fits (`ok=True`). Minimal example ~20 lines.
- Reference for `AdmissionResult` is **generated from its docstring** (see I-002).

**Code ref:** `src/uxok/core/_core.py:291`; `src/uxok/core/_core_facet.py`

#### C-006 — Capability events `core.capability.revoked` and `core.capability.rebound` are undocumented
**File:** `src/uxok/core/_core_facet.py` (event emission)
**Signal:** `Undocumented public interface`

Two new events a plugin can subscribe to are absent from the docs:
- `core.capability.revoked` — payload `{capability, old_provider_id}`
- `core.capability.rebound` — payload `{capability, old_provider_id, new_provider_id}`

These fire when a provider behind a capability is revoked or swapped, which is exactly the hot-reload context. `how-to-use-hot-reload.md` already teaches subscribing to `core.plugin_reloaded` and `core.plugin_error` with their payloads, so the new events have an established home and pattern there.

**Expected location per skill:**
- UPDATE `docs/how-to/how-to-use-hot-reload.md` — **Diátaxis: How-to**, 200–600 words. Add a step (matching the existing `core.plugin_reloaded` step) showing `@event("core.capability.revoked")` and `@event("core.capability.rebound")` handlers with their exact payload keys. Keep it tight — the page is already near the density ceiling, so add only the subscription steps and link to the explanation page for *why* a rebind occurs.
- Cross-link from `explanation/capability-system.md` (M-001) where it already discusses provider swap on reload.

**Code ref:** `src/uxok/core/_core_facet.py`

#### M-001 — `explanation/capability-system.md` describes only open-mode; `requires` gained a second runtime role it predates
**File:** `docs/explanation/capability-system.md`
**Signal:** `Version mismatch — materially incomplete`

The page is accurate for the default `open` posture but now incomplete: `requires` quietly gained a second role — in `declared`/`sealed` mode it (with `resolves`) gates **runtime** resolution, not just load order. The page's "What plugins declare" section frames `requires` purely as a startup precondition (lines 16–22). A reader in a non-default mode following this page would not anticipate that resolving an undeclared capability now raises `CapabilityAccessError`. This is incompleteness, not inaccuracy — the default behaviour described is still correct — hence Medium, not High.

The fix must respect the public non-restatement rule: this page should add a short note that `requires` has a runtime role under the secure postures and **link** to `explanation/secure-capability-access.md` (C-001) for the full model, rather than restating the per-mode "why" here. Also add a one-line cross-reference to the two new capability events (C-006) in the existing reload/swap paragraph (line 61).

**Doc ref:** `docs/explanation/capability-system.md:16-22, 61`
**Code ref:** `src/uxok/plugin/_base.py:75-154`; `src/uxok/core/_core_facet.py`

#### I-001 — Pre-existing RFC-citation leaks point into the build-excluded `proposals/` tree

The `docs/proposals/` tree is excluded from the MkDocs build, so any "RFC 0001 §3.2.2"-style citation rendered into the public site is a dangling pointer. Existing leaks:

- **Public page (in scope to fix):** `docs/how-to/how-to-use-plugin-collections.md:148` — the prose carries `(RFC 0001 §3.2.2: discovery must not be a backdoor to invocation)`. This should be stripped and reworded to plain English as part of the C-007 update to this page.
- **Docstrings (code change — NOT scoped here):** `src/uxok/core/_core_facet.py` (lines 2, 23, 28, 53, 60, 85–87, 122, 133, 138, 148) and `src/uxok/registry/_plugin_proxy.py` (lines 5, 61, 214) cite RFC numbers and section markers. These render verbatim into the **generated** reference pages, so they leak the same dangling pointers onto the site. Fixing them is a source-docstring edit (a code change) and is recorded here only as a notice — it is out of the writer's scope and must not be folded into a public-page task.

All recommended new and updated pages in this handoff must describe API and behaviour in plain English only — no RFC numbers, no section markers.

#### I-002 — The new types reach the website only through docstrings (generated reference)

`CapabilityAccessError`, `StalePluginError`, `AdmissionResult`, `CapabilityFacet`, `CoreFacet`, the `resolves` parameter, and `Core.check_plugin` appear in the public `reference/` section **only** if their source docstrings are complete; the writer never authors `reference/` `.md` files. Two consequences for this work: (1) the explanation/how-to pages are the only authored surface for these types, so the coverage gap is real even where a generated reference page exists; (2) some of the relevant docstrings currently carry RFC citations (I-001), so the generated reference quality is itself degraded until the source is fixed. `mkdocs build --strict` was not run in this environment (toolchain not installed), so the generated output was not inspected live.

#### I-003 — `kernel.dispatch` reserved grant noted for completeness

The directive notes a second reserved grant, `kernel.dispatch`, which (with `requires | resolves`) is part of what a plugin may resolve in declared/sealed mode. It is peripheral to the page plan above and need not be taught on its own, but the lock-down how-to (C-001) or the secure-access explanation (C-001) should at least name it alongside `kernel.lifecycle` so the set of reserved grants is not presented as a single special case. Recorded as an ambient completeness note, not an action item of its own.
